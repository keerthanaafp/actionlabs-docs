Action Labs — Web App Documentation

Reference for every web view in the Action Labs Fieldproxy app. Covers what each screen does, the filters and sections it exposes, and how it connects to the rest of the workflow.

Overview

Action Labs runs a field service operation across multiple clusters, with field engineers (FEs) resolving customer tickets on a billing model tied to shift hours and per-hour rates. The Fieldproxy build is split into three web modules:

  • Action Labs Web App — the billing dashboard (Project Metrics) used by finance to see what's billable to the project versus what's billable to Action Labs.
  • Project Management — a 30-day rolling operations dashboard showing resolved-ticket counts, FE hours, and amount summaries per project.
  • Field Service Management (FSM) — the day-to-day team-leader tool: dashboards, job board, schedule/dispatch, jobs reports, SLA report, workforce screens (skill mapping, time-off), and the customer / contact screens.

Status legend

Several views (Main Dashboard, Jobs, Jobs Report, Project Metrics) bucket tickets by status. The four statuses below are treated as "billable / completed" everywhere a ticket count is shown.

Module

Action Labs Web App

The billing dashboard — used by finance to slice billable hours by project, employee, and date.

Project Metrics

Web App / project_metrics.json  ·  homescreen.json (same view, two filenames)

The single billing dashboard for Action Labs. Combines completed-ticket data (tickets_management) with attendance (attendance_management) and the per-project rate card (project_billing) to compute three numbers: billable to project, billable to Action Labs (idle hours during shift), and total amount.

Purpose

Finance opens this once a billing cycle to reconcile what each FE earned their project versus the hours Action Labs covered when the FE was on shift but not on a ticket.

Filters

  • Project Name (dropdown)
  • Cluster (text)
  • Employee Name (dropdown)
  • Ticket Completed Date — from / to
  • Apply / Clear buttons

Sections on the page

  • KPI cards — Total Resolved Tickets, Total FEs, Total Hours, Billable to Project, Billable to Action Labs, Total Amount.
  • Breakdown Per Project — bar chart, top projects by billable amount.
  • Billable Per Day — line chart of daily billing.
  • Billable to Project table — per-ticket detail (employee, project, ticket #, in-transit time, completed-at, total hours, amount, shift start/end).
  • Billable to Action Labs table — per-day attendance vs ticket hours, with the idle-hour amount Action Labs absorbs.

Key interactions

  • Apply / Clear refresh every KPI, table, and chart in one chained update_data action.
  • Tables expose pagination + search; rows are read-only.

Module

Project Management

The 30-day rolling operations summary — same numbers as Project Metrics, but always scoped to the last 30 days.

Dashboards

Project Management / dashboards.json

A snapshot dashboard of resolved-ticket activity over the trailing 30 days. No filters — it always shows the same window — so it works as a "what's happening this month" landing page.

Purpose

Quick health check for ops leads: who delivered tickets this month, how much it cost, and which projects pulled the most engineering hours.

Filters

  • None — fixed last-30-days window.

Sections on the page

  • KPI cards — Total Resolved Tickets, Total FEs, Total Hours, Billable to Project, Billable to Action Labs, Total Amount.
  • Billable to Project table — large detail table, paginated, searchable.
  • Billable to Action Labs table — paginated daily attendance breakdown.
  • Bar chart — top 10 projects by billable amount.

Key interactions

  • In-table search and pagination.
  • Sidebar navigation between PM views.

Project Management Container

Project Management / project_management.json

A master container file that wires the Project Management module's sidebar to its child views. Not a screen users land on directly — it's the shell.

Purpose

Defines the navigation tree for the PM module so the Fieldproxy runtime knows which views show up in the sidebar and where each link goes.

Notes

  • If you add a new view to Project Management, register its app_id / view_id here so the sidebar links resolve.
  • Style changes to the sidebar (header, color, active-state) live in this file.

Module

Field Service Management

The team-leader tool — daily dispatch, ticket triage, reports, and customer screens. The largest module by view count.

Main Dashboard

Field Service Management / Web App / main_dashboard.json

The home screen a team leader lands on. A filter bar at the top, a wall of status KPI counts, four "live" ticket lists for what's happening right now, and three charts at the bottom for slicing the same dataset.

Filters

One filter row drives the whole page. Pick the values you want and click Apply Filters — every KPI count, every Live Job Status list, and every analytics chart re-runs against the new window in a single chained update_data action.

  • Cluster — dropdown, defaults to All Clusters.
  • From Date — defaults to today minus 30 days.
  • To Date — defaults to today.
  • Apply Filters — the button that fans the new filter values out to every widget on the page.

KPI cards

Fourteen cards across the top of the dashboard. Each card's number is a count of tickets matching that bucket inside the active filter window.

Click-through behaviour. Clicking any status KPI card navigates to the Jobs list with that status pre-applied as a filter argument — only those tickets show up on the destination page, so the dashboard count and the list you land on are always the same set. The three "metric" cards (Total Tickets, Avg Visit Time, Pending Visits) are read-only and don't navigate.

Live Job Status

Four side-by-side lists below the KPI strip, one for each in-flight state. Each row in a list is a single ticket: the ticket ID and a coloured priority badge — Urgent, High, Medium, or Low. Clicking a row opens the Job Detail View for that ticket. Lists with nothing in them show a "No data" empty state.

  • ETA Updated — engineer has acknowledged and put down an ETA, but hasn't moved yet.
  • Engineer in Transit — FE is on the road to the site.
  • Arrival on Site — FE has reached the customer.
  • Job Start — work is in progress.

Analytics

Three charts at the bottom of the page. They share the same filter window as everything above — change a filter and they re-render along with the rest.

  • SLA Requests — donut chart of tickets by SLA status (within / breached / at risk). Source: sla_status_data.
  • Types of Request — bar chart of ticket counts by request type (install / repair / preventive / etc.). Source: type_of_request_data.
  • Priority Levels — bar chart of ticket counts split by Urgent / High / Medium / Low. Source: priority_chart_data.

Dashboard

Field Service Management / Web App / dashboard.json

A secondary, more visualization-heavy dashboard. Where Main Dashboard focuses on counts, this one leans on charts and a map of FE activity.

Purpose

Visual snapshot for week-in-review meetings — open ticket count, jobs completed, active engineers, average visit time, plus category mix and live FE locations.

Sections on the page

  • KPI cards — Open Tickets, Jobs Completed, Active Engineers, Avg Visit Time.
  • Status bar chart — ticket counts per status.
  • Category donut — share of tickets by category.
  • Activity map — FE pins keyed by latest activity.
  • Activity list — paginated companion to the map (click to focus a marker).
  • Jobs table — recent tickets with search and pagination.

Key interactions

  • Click an item in the activity list to highlight its marker on the map.
  • Pagination on both jobs table and activity list.

Schedule / Dispatch Board

Field Service Management / Web App / schedule.json

The weekly dispatch board. Technicians on the Y-axis, days of the week on the X-axis, and a pool of unscheduled jobs alongside. Drag-and-drop a job card onto a technician-day cell to assign it.

Purpose

The primary scheduling tool. Replaces the old "WhatsApp + spreadsheet" loop with a visual board so leads can see capacity at a glance.

Filters

  • Cluster (dropdown)
  • Employee (dropdown)
  • Schedule stats summary at the top

Key interactions

  • Drag job card → technician × date cell to assign.
  • Click a job card to view or edit its details.
  • Unscheduled-jobs pool drains as you assign.
Layout note: the sticky-filter pattern with dropdowns is known not to render correctly on this view — keep filters non-sticky.

Location Intelligence

Field Service Management / Web App / location_intelligence.json

A pivot of clusters × ticket statuses. Each cell is a count, so you can see at a glance which cluster has the most cancelled tickets, the most on-hold tickets, etc.

Purpose

Cross-cluster comparison. Useful when a regional manager wants to know which cluster is under-performing on resolution or has unusually high cancellations.

Filters

  • Cluster dropdown
  • "Search by Cluster" text input

Sections on the page

  • Cluster × Status pivot table — clusters as rows, statuses as columns, ticket counts in each cell.

Jobs List

Field Service Management / Web App / jobs.json

The day-to-day ticket board. A paginated table with rich filters across status, priority, contact, employee, category, and service type.

Purpose

Where team leaders triage. Filter to "P1 + Open" first thing in the morning to see what needs an FE assigned now.

Filters

  • Date range (defaults from default_dates)
  • Status, Priority, Category, Service Type (dropdowns)
  • Contact, Contact Name, Employee (dropdowns)
  • KPI strip with quick-filter counts

Sections on the page

  • KPI strip — counts by major status.
  • Jobs table — paginated, searchable, with row-action buttons.

Key interactions

  • Click a row to open Job Detail View.
  • Edit button → Edit Job.

Jobs All Data

Field Service Management / Web App / jobs_all_data.json

A "raw" jobs view — every column from tickets_management surfaced as a wide, scrollable table. Designed for power users who need to filter on a column the standard Jobs view doesn't expose.

Purpose

Search-by-anything escape hatch when the Jobs List filters aren't enough, and source-of-truth for ad-hoc CSV exports.

Filters

  • "Search by …" dropdown — pick a column (Ticket ID, Hospital, Employee, etc.)
  • Search text input
  • Search / Clear buttons
Canonical pattern. This file is the canonical reference for the dropdown-driven search pattern: a "Search By" dropdown whose values match a SQL CASE ladder byte-for-byte, plus a text input and Search/Clear buttons that call update_data on the table and its count query. Other client builds clone this view's row_search block.

Key interactions

  • Heavy-table pagination trio (pageSize: 25 / min_lines: 10 / page_size: 100) — required because the underlying table has thousands of rows.
  • CSV export from the actions row.

Job Detail View

Field Service Management / Web App / jobs_detailed_view.json

Full read-out for a single ticket. Pulls together job metadata, the FE activity log, expense receipts, the on-site checklist, and the job location on a map.

Purpose

What a team leader opens when investigating a specific ticket — usually because of a customer complaint, an SLA breach, or to verify expense claims before approving them.

Sections on the page

  • Header — ticket title and #TICKET_ID.
  • Job Details — customer, address, priority, status, ETAs, employee assigned.
  • Visit data — every check-in / check-out for the ticket.
  • Activity log — chronological events.
  • Expense data — receipts uploaded by the FE.
  • Checklist — completed checklist items.
  • Job location map — pin at the customer site.

Key interactions

  • Edit action → Edit Job.
  • Pagination on activity-log, visit, expense, and checklist sub-tables.

Edit Job

Field Service Management / Web App / edit_job.json

The form view for editing a ticket. Loads the current values into inputs (location, ETA, employee, priority, status, etc.) and writes back via an UPDATE query.

Purpose

Lets a team leader fix or update ticket fields after creation — re-assign engineer, push an ETA, change priority, correct address.

Sections on the page

  • Header with the ticket title.
  • Form fields grouped by category (location, scheduling, assignment, status).
  • Save / Cancel actions.
Save behavior: uses the COALESCE(NULLIF(input,''), col) "preserve unchanged on edit" pattern, with explicit casts (::timestamptz, ::time, etc.) per column type so date / time / numeric fields don't silently fail.

Jobs Report

Field Service Management / Web App / jobs_report.json

Multi-pivot report screen. Slices the same job dataset by date, employee, company, and contact — each as its own table — so you can answer "who handled the most jobs this month?" and "which company gave us the most tickets?" in one place.

Purpose

Monthly / quarterly review. Often exported and pasted into a deck.

Filters

  • Date range, Cluster, Status, Priority, Employee, Customer (combined apply button)

Sections on the page

  • Date breakdown table.
  • Employee breakdown table.
  • Company breakdown table.
  • Contact breakdown table.
  • Jobs table — flat list under the breakdowns.

Key interactions

  • Per-table CSV export.
  • Pagination on every breakdown table.

SLA Report

Field Service Management / Web App / sla_report.json

Customer-wise SLA compliance: response time, resolution time, and breach counts compared against contractual targets.

Purpose

What account managers send to customers ahead of the QBR — proof of SLA performance, broken down by company.

Sections on the page

  • SLA Report — Customer Wise table (paginated).
  • Record count and CSV export in the actions row.

Key interactions

  • "Total: N records" counter that updates with the table data.
  • CSV export action.

Skill Set Mapping

Field Service Management / Web App / skill_set_mapping.json

Maps each FE to the skill sets they're certified for, plus their latest location and shift schedule. Lets a dispatcher pick the right person for a specialized job.

Purpose

When a P1 ticket comes in for a specialized machine, this is where the dispatcher answers "who in this cluster is qualified and on shift?".

Sections on the page

  • Skill map — engineer rows × skill columns.
  • Skill list — flat list of (engineer, skill) pairs.
  • Latest Location — last known FE location.
  • Schedule — engineer shift overview.

Time Off

Field Service Management / Web App / time_off.json

Time-off requests for field engineers. Lists each request with its dates, type, and approval status.

Purpose

So the dispatcher knows who is unavailable when scheduling. Also the audit trail for HR.

Sections on the page

  • Time Off List — table of requests, paginated.

Module

Customer

The customer book. Two parallel hierarchies — companies and contacts — each with a list view and a single-record detail view.

Companies

Field Service Management / Web App / customer / company.json

Master list of customer companies. Each row is one company; click into it to open Company Detail.

Sections on the page

  • Companies table — paginated, searchable.
  • Total count actions row.

Key interactions

  • Row click → Company Detail.

Company Detail

Field Service Management / Web App / customer / company_detail_view.json

Single-company view. KPIs at the top, then the company's contacts, the jobs opened against it, visit history, expenses, and checklists.

Sections on the page

  • Company KPI — counts (total jobs, open, completed, etc.).
  • Company details — name, address, primary contact, status.
  • Contacts — table of people at the company.
  • Jobs — every ticket opened against the company.
  • Jobs by category — chart.
  • Visit / Expense / Checklist — operational drill-downs.

Key interactions

  • Click a job row → Job Detail.
  • Click a contact row → Contact Detail.

Contacts

Field Service Management / Web App / customer / contact.json

Master list of contacts (people) across all customer companies. The counterpart to the Companies list.

Sections on the page

  • Contacts table — paginated, searchable.
  • Total count actions row.

Key interactions

  • Row click → Contact Detail.

Contact Detail

Field Service Management / Web App / customer / contact_detailed_view.json

Single-contact view — the same operational drill-downs as Company Detail but scoped to the tickets the contact personally raised.

Sections on the page

  • Contact KPI — counts.
  • Contact details — name, role, phone, email, parent company.
  • Jobs — tickets raised by this contact.
  • Jobs by category — chart.
  • Visit / Expense / Checklist — operational drill-downs.

Container

FSM Master Container

Field Service Management / Web App / field_service_management.json

Master container — labelled Team Leader Dashboard internally. Defines the FSM sidebar, the routes between every FSM view, and the schedule stub. Edit this when you add or rename a FSM screen.

Notes

  • Sidebar order, icons, and active-state colors live in this file.
  • Adding a new FSM view? Register its app_id + view_id here so the sidebar links pick it up.
  • The schedule stub here is superseded by schedule.json — keep edits there.
